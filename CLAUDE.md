## Project: Server Foundry

A web platform that lets non-technical users host their own multiplayer game servers on hardware they already own (home PC, spare server, rented VPS). Users run one install command, and the platform handles deployment, monitoring, updates, and management.

**Key distinction** — this is NOT a managed game server host. Users bring their own hardware. We are the orchestration and management layer on top.

**Self-hosted on Unraid** behind Cloudflare Tunnel. Public domain, public-facing dashboard.

## Critical rules — never violate

1. **Never invent data.** If a value isn't in the database or computable from real data, render `—` (em-dash) and leave a `// TODO:` comment. Do not fabricate metrics, timestamps, user names, or anything else.

2. **Never modify working production code without an explicit plan.** When asked to "fix" or "improve" something, first explain what you'll change and why, then wait for approval. Surgical edits over rewrites — every time.

3. **Always preserve the user's working state.** Before destructive operations (deleting files, restructuring directories, dropping tables), ask. After every meaningful change, verify `npm run build` and `npm run lint` still pass.

4. **Never commit secrets.** No API keys, no passwords, no tokens, no `.env` contents in code or commits. `.env.example` shows shape, `.env.local` is gitignored.

5. **Never add packages casually.** If you think a new dependency is needed, justify it. Prefer the standard library, the existing stack, and copy-pasteable patterns over new dependencies.

6. **Never add Anthropic / Claude attribution to commits.** Use clean conventional commit messages. No "Generated by Claude" lines.

7. **Always read the relevant docs before coding.** If you're touching the data layer, read `docs/data-model.md` first. If you're touching auth, read `docs/security.md` first. The docs encode decisions that aren't visible in the code.

8. **Match existing patterns.** When adding to a file or module, look at how the surrounding code handles imports, error handling, naming, types — and match it.

9. **Test the critical path manually after major changes.** The critical path right now is: landing page loads → waitlist signup works → email confirmation arrives. As we expand, this critical path will grow.

## Development workflow

Local development uses Bun (preferred — faster install + test runner). The Docker
production image and CI pipeline use npm + Node 22 — `package.json` scripts work
with both runtimes.

```bash
# Setup (local dev)
bun install
cp .env.example .env.local  # then fill in values
docker compose up -d postgres
bun run db:generate         # create migration SQL from schema.ts
bun run db:migrate          # apply Drizzle migrations
bun run dev                 # start Next.js on http://localhost:3000

# Quality gates (must pass before commit)
bun run lint
bun run typecheck
bun run test
bun run build

# Database
bun run db:studio           # browse data via Drizzle Studio

# Docker (for Unraid deployment) — uses npm + Node 22 inside the image
docker build -t server-foundry .
docker run -p 3000:3000 --env-file .env.local server-foundry
```

If Bun is not installed, `npm install` and `npm run …` work identically.

## How to use this document set effectively

1. Read `CLAUDE.md` (this file)
2. Read `docs/roadmap.md` to understand where the project is in its lifecycle
3. Read the docs relevant to your task (e.g. `docs/data-model.md` for database changes, `docs/api.md` for endpoints)
4. Check `docs/conventions.md` if you're unsure how to structure code
5. Reference `docs/security.md` whenever auth, secrets, or user data is involved
